向后兼容性策略
==============================

.. versionadded: 6.0

Pytest 是一个积极发展的项目，已经发展了数十年。
我们不断了解新的和更好的结构来表达测试的不同细节。

在实施这些修改时，我们试图确保平稳过渡，不希望对用户和社区/插件作者施加不必要的变动。

目前，pytest 考虑多种类型的向后兼容性过渡：

a) 平凡的：可以轻松转换为新机制且不会引起问题的 API。

   我们试图无限期地支持这些，同时通过文档鼓励用户切换到更新或更好的机制。

b) 过渡性的：旧 API 和新 API 不冲突，我们可以通过使用警告帮助用户过渡，同时在长时间内支持两者。

   我们只在主要版本（例如，如果我们在 3.0 中弃用某些东西，我们将在 4.0 中开始移除它），并在至少两个小版本中保留它（例如，如果我们在 3.9 中弃用某些东西而 4.0 是下一个版本，我们在 5.0 而不是 4.0 中开始移除它）。

   计划在主要版本 X 中移除的弃用功能将使用警告类 `PytestRemovedInXWarning`（:class:`~pytest.PytestDeprecationWarning` 的一个子类）。

   当弃用到期时（例如，4.0 发布时），我们不会立即移除弃用的功能，而是使用标准警告过滤器将 `PytestRemovedInXWarning`（例如 `PytestRemovedIn4Warning`）默认转变为**错误**。这种方法明确表明移除即将到来，并且仍然给你时间将弃用的功能转变为警告而不是错误，以便它可以在你自己的时间处理。在下一个小版本（例如 4.1）中，该功能将被有效移除。

c) 真正的破坏性变更只有在正常过渡不合理地不可持续且会使重要的开发或功能推迟数年时才应考虑。此外，它们应该限于实际用户数量非常少的 API（例如，只影响某些插件），并可以提前与社区协调。

   此类即将到来的变更示例：

   * 移除 ``pytest_runtest_protocol/nextitem`` - :issue:`895`
   * 重新排列节点树以包含 ``FunctionDefinition``
   * 重新排列 ``SetupState`` :issue:`895`

   真正的破坏性变更必须首先在一个 issue 中宣布，包含：

   * 变更的详细描述
   * 基本原理
   * 对用户和插件作者的预期影响（示例见 :issue:`895`）

   在 issue 上没有硬性 *-1* 后，它应该跟一个初始的概念验证 Pull Request。

   这个 POC 作为评估影响和潜在灵感的协调点，以想出一个过渡解决方案。

   合理的时间后，PR 可以合并以构建新的主要版本。

   要使 PR 从 POC 成熟为接受，它必须包含：
   * 设置弃用错误/警告，帮助用户修复和移植他们的代码。如果可以在当前系列下引入弃用期，在真正的破坏性变更之前，它应该在单独的 PR 中引入并成为当前发布流的一部分。
   * 详细描述基本原理和如何移植代码的示例在 ``doc/en/deprecations.rst`` 中。


历史
=========


主要关注平稳过渡 - 立场（6.0 之前）
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

保持向后兼容性在 pytest 项目中具有非常高的优先级。尽管我们多年来已经弃用了功能，但大多数功能仍然得到支持。pytest 中的所有弃用都是因为出现了更简单或更有效的方式来完成相同的任务，使旧的方式变得不必要。

随着 pytest 3.0 的发布，我们引入了一个清晰的沟通方案，用于说明我们何时会实际移除旧的、有问题的功能，并礼貌地要求你使用新的好东西，同时给你足够的时间来调整你的测试，如果有充分的理由保留弃用功能的话。

为了传达变更，我们使用自定义警告层次结构发出弃用警告（参见 :ref:`internal-warnings`）。这些警告可以使用标准方式抑制：:option:`-W` 命令行标志或 :confval:`filterwarnings` 配置选项（参见 :ref:`warnings`），但我们建议谨慎且临时地使用这些，并在可能时听从警告。

我们只在主要版本开始移除弃用功能（例如，如果我们在 3.0 中弃用某些东西，我们将在 4.0 中开始移除它），并在至少两个小版本中保留它（例如，如果我们在 3.9 中弃用某些东西而 4.0 是下一个版本，我们在 5.0 而不是 4.0 中开始移除它）。

当弃用到期时（例如，4.0 发布时），我们不会立即移除弃用的功能，而是使用标准警告过滤器将它们默认转变为**错误**。这种方法明确表明移除即将到来，并且仍然给你时间将弃用的功能转变为警告而不是错误，以便它可以在你自己的时间处理。在下一个小版本（例如 4.1）中，该功能将被有效移除。


弃用路线图
-------------------

目前弃用并在先前版本中移除的功能可以在 :ref:`deprecations` 中找到。

我们使用里程碑以及 GitHub 上的 `deprecation <https://github.com/pytest-dev/pytest/issues?q=label%3A%22type%3A+deprecation%22>`_ 和 `removal <https://github.com/pytest-dev/pytest/labels/type%3A%20removal>`_ 标签来跟踪功能的未来弃用和移除。
